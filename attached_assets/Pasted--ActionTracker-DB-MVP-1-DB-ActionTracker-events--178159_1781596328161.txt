● ActionTracker DB設計書（MVP版） 
● 1. DB設計方針 
● ActionTrackerでは、生ログを保存する events テーブルを中心とした構成を採用する。 
● 分析用データは生ログから生成し、表示速度向上のために sessions および 
transitions テーブルへ集約する。 
● MVPではテーブル数を最小限に抑え、将来的なAI分析やクラウド同期に対応可能な構
造とする。

● 3. テーブル定義 
● events 
● 概要 
● 生ログ保存テーブル。 
● ActionTrackerにおいて最も重要なテーブル。 
● ウィンドウ変更やアイドル状態など、発生したイベントをそのまま保存する。

● 4. インデックス設計 
● events 
● CREATE INDEX idx_events_timestamp 
● ON events(timestamp); 
● CREATE INDEX idx_events_service 
● ON events(service); 
● CREATE INDEX idx_events_category 
● ON events(category); 
● sessions 
● CREATE INDEX idx_sessions_start_time 
● ON sessions(start_time); 
● CREATE INDEX idx_sessions_service 
● ON sessions(service); 
● transitions 
● CREATE INDEX idx_transitions_timestamp 
● ON transitions(timestamp); 
● CREATE INDEX idx_transitions_services 
● ON transitions(from_service, to_service); 
● 5. データ保持方針 
● events 
● 保持期間 
● 無期限 
● 生ログとして保存する。 
● sessions 
● 保持期間 
● 無期限 
● 分析用データとして保持する。 
● transitions 
● 保持期間 
● 無期限 
● 行動分析に利用する。 
● category_rules 
● 保持期間 
● 無期限 
● ユーザーによる編集を許可する。
